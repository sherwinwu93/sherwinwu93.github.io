Docker

Docker是什么?

Docker是Linux容器,可以理解成轻便的Linux虚拟机,只有需要的应用及其依赖.

什么端口?

使用Docker-compose, 选择image生成实例, 由容器内部的接口映射到容器外部(也就是服务器的接口), 并且提供相应环境变量.
例如 某个微服务 在容器内部是8080, 映射到服务器上是8081, 环境变量指定是正式环境

为什么要使用Docker?

  1. 构造和部署管道, 只需要一个命令就可以构造和部署到任何环境
  2. 把微服务的基础设施变成可以执行和管理的代码去对待, Dockerfile和run.sh
  3. 一旦创建了微服务镜像,确保它部署后永远不会改变
  4. 如果在服务器上运行,服务器自身容易发生环境变量的改变,确保其不会发生改变

Docker-compose

Docker-compose,用来简化Docker的操作
编写yml文件,使用镜像,指定容器内外端口(映射)和环境变量,管理实例(加载/卸载,运行,停止,删除)

分库分表

什么是分库分表

分库分表,指把一张表水平分割,分为多个子表,放入不同的数据库内

如何切分库分表

水平分库分表: 表结构不变,按每条数据分割,把表分成多个子表,放入或不放入不同的库中
记录数过多
垂直分库分表: 表的结构改变,按字段分割.
一条记录的数据太多

为什么要分库分表

磁盘, CPU, 内存, 宽带, 连接数不够用了

切分策略

  1. 范围,根据某字段的范围,比如ID,每百万放一张表
    缺陷: 写入数据都是写新表,新表写入问题比较大
  2. Hash切分, 用ID对总表数取余数, 决定库和表
    解决了新表写入问题
    缺陷: Hash算法如果不合理的话,会造成数据分布不均匀的情况. 还有跨表查询和分页的问题
  3. 映射表
    把每行数据的ID和分表映射记录下来

分库分表产生的问题

  1. 查询数据难度增加,需要先查询映射表,再查询到数据
  2. 分布式ID. UUID
  3. 分布式事务, 之前的本地事务没法用了
    1. 在业务里避免分布式事务
    2. 用分布式框架
  4. 跨表关联/分页/排序: 在业务代码中分别查询,然后组装数据 Sharding-Proxy
  5. 业务服务器直连, 数据库(代理). 都是把逻辑抽象封装,业务无需关心细节

分库分表如何落地?

Mybatis

Mybatis我们项目用的是MybatisGenerator,尽量不写SQL,在Java代码中解决.
比如:用一张新表生成统计数据,定时执行.或者直接把数据存入Elasticsearch中,解决搜索和统计的问题.

一级缓存和二级缓存的区别

一级缓存范围在一个SqlSession内部,不是全局缓存
二级缓存是全局
如果二级缓存开启的话,执行顺序是先去二级缓存查询,二级缓存有,则直接返回.没有再去一级缓存查,有直接返回.没有的话,去数据库查.
多表查询的时候,多个sqlSession的时候,另外一个事务未提交的时候,二级缓存不生效.
一个事务已经提交的时候,二级缓存生效有50%的概率.
使用微服务框架的话,二级缓存和一级缓存的变量都可能不在同一台服务器上,容易出现脏读~幻读的现象.不建议使用.

GC

GCRoot

GCRoot指的是栈中,本地方法栈,方法区.
直接引用的对象,或者间接引用的对象.
没有被引用则会被删除

GC算法

  1. 标记-清理: 标记后直接清理
    缺: 内存碎片
  2. 标记-整理: 标记后删除,再前移动
    缺: 移动频繁,开销大
  3. 复制: 复制到另一块区域
    缺: 内存开销大

实际GC

分为新生区,和老年区
新生区Young: 分为Edan, Survivor0, Survivor1. 因为很多对象生命周期短,所以是8:1:1
老年区Old: Old. 大对象直接出生在Old区,比如大数组.

实际GC:
新创建的对象实例,在Edan.然后Young GC. 一次Edan+S1复制到S0.再一次GC由Edan+S0复制到S1.每一次复制,年龄+1.
到Age=6的时候,复制到Old.
Old满了,产生FullGC.用的是标记清理或标记整理的算法.

  • 如果Edan+S1复制到S0满了,直接进入Old区

Java

Java的多态是什么?

编译类型是一个类,实际类型可能不同(可以是当前类,也可以子类),实际会调用实际类型的方法,造成方法行为的不同.

Java中变量,代码块,构造器之间的执行顺序是怎样的?

先父子静态变量;再父子静态代码块;先父普通成员变量+代码块+构造器;再子普通成员变量+代码块+构造器.

final关键字有什么作用?

  • 用于类时:类不能被继承,且所有方法也被final修饰
  • 用于方法:方法不能被子类重写
  • 用于变量:变量一旦初始化,不能被修改

Integer类会进行缓存吗?

Integer 在-128到127之间,有一个共有的缓存数组. 所以会出现-128到127之间,会相等.

抽象类有那些特点?

  1. 可以有抽象方法,由子类来实现
  2. 不能被实例化
  3. 抽象类可以被继承,子类没实现,则继续作为抽象类

String, StringBuffer和StringBuilder的区别是什么?

String: 不可变,用final修饰,每次变化都会产生新的对象
StringBuilder和StringBuffer: 可变,StringBuffer读写用了synchronized修饰,线程安全.StringBuilder不安全.

hashCode()和equals()

  • hashCode()
    • 对对象做散列,比如: HashMap
    • 快速判断对象是否equals()
  • 为什么hashCode()和equals()方法一起重写?
    • 只重写hashCode,会导致相等的对象但内存地址不同,HashMap无法去重
    • 只重写equals方法,equals的对象HashCode不同,进入不同的数组,仍然无法去重

clone()方法

返回对象的副本,相等,但是存储位置不同

toString()方法

返回对象+@+hashCode的十六进制

gc调用的,有可能重新被引用,也可能被清理

编译型,解释型,和伪编译型语言的区别?

  • 编译型: 根据平台编译成机器码,打包成可执行程序.如: C,C++
  • 解释型: JavaScript, 对源码逐行解释且立即执行

Redis

1.Redis是什么?

又基于内存,又持久化的键值对数据库.

2.Redis过期key是怎么样清理的?

  1. 惰性清理: 查询时发现过期了,将key删除
  2. 定时清理: 周期性地清理所有DB,从DB的expires字典(key对象,value过期时间)随机取出20个key,删除掉过期的.
    如果超过25%的话,就对这个db清理.没有就下一个.
  3. 内存不够时清理:
    1. 不处理,写入时报错
    2. 从所有key中挑选,进行淘汰.
      1. 随机挑选进行淘汰
      2. 根据使用时间最远的淘汰
      3. 使用频率最低的进行淘汰
    3. 从设置了过期时间的key中,进行淘汰.

4.Redis的性能为什么这么高?

  1. 完全基于内存,没有磁盘IO,速度快
  2. 单线程,没有上下文的切换开销,不用考虑加锁释放锁,也没有死锁
  3. 数据结构专门设计的
  4. 监听多个IO流,空闲时把线程阻塞掉.有事件时,把阻塞唤醒,轮询发出事件的流,按顺序处理流.
    单个线程处理多个连接请求,减少网络IO.

8.如何解决Redis缓存穿透问题?

故意请求一些Redis不存在的key,请求打到数据库上,导致数据库压力过大.

  1. 对传参进行校验,避免一些无效redis请求
  2. 对于找不到的key,设置个短期过期时间
  3. 把所有可能存在的数据放入到大的map中,通过map进行过滤

9.如何解决Redis缓存击穿问题?

某个热点key过期,导致大量请求转向数据库,数据库压力过大

  1. 热点key设置永不过期
  2. 对于热点数据加锁(加在数据库上),每次只能取一次,每次等待100ms.

10.如何解决Redis缓存雪崩问题?

短期内大量key失效,所有请求转向数据库

  1. 给缓存失效时间加随机值,避免集体失效
  2. 搞两个缓存,缓存B比缓存A长一点,如果缓存A失效了,去缓存B找数据,然后用个异步线程来更新缓存A

11.如何解决缓存与数据库的数据一致性问题?

原因:

  1. 多个写请求的执行顺序不同导致脏数据.
  2. 先读了,再更新
  3. 数据库读写分离的,从库需要时间一定时间才能更新

错误解决方案:

  1. 先更新数据库,再更新缓存
  2. 先删除缓存,再更新数据
  3. 先更新数据库,再删除缓存

正确方案:

  1. 写请求之前要先获取分布式锁,才能更新,效率变低了
  2. 先更新数据库,异步删除缓存,删除失败后重试
