Docker

Docker是什么?

Docker是Linux容器,可以理解成轻便的Linux虚拟机,只有需要的应用及其依赖.

为什么要使用Docker?

  1. 构造和部署管道, 只需要一个命令就可以构造和部署到任何环境
  2. 把微服务的基础设施变成可以执行和管理的代码去对待, Dockerfile和run.sh
  3. 一旦创建了微服务镜像,确保它部署后永远不会改变
  4. 如果在服务器上运行,服务器自身容易发生环境变量的改变,确保其不会发生改变

Docker-compose

Docker-compose,用来简化Docker的操作
编写yml文件,使用镜像,指定容器内外端口(映射)和环境变量,管理实例(加载/卸载,运行,停止,删除)

Docker-compose的使用?

使用Docker-compose, 选择image生成实例, 由容器内部的接口映射到容器外部(也就是服务器的接口), 并且提供相应环境变量.
例如 某个微服务 在容器内部是8080, 映射到服务器上是8081, 环境变量指定是正式环境

分库分表

什么是分库分表

分库分表,指把一张表水平分割,分为多个子表,放入不同的数据库内

如何切分库分表

水平分库分表: 表结构不变,按每条数据分割,把表分成多个子表,放入或不放入不同的库中
记录数过多
垂直分库分表: 表的结构改变,按字段分割.
一条记录的数据太多

为什么要分库分表

磁盘, CPU, 内存, 宽带, 连接数不够用了

切分策略

  1. 范围,根据某字段的范围,比如ID,每百万放一张表
    缺陷: 写入数据都是写新表,新表写入问题比较大
  2. Hash切分, 用ID对总表数取余数, 决定库和表
    解决了新表写入问题
    缺陷: Hash算法如果不合理的话,会造成数据分布不均匀的情况. 还有跨表查询和分页的问题
  3. 映射表
    把每行数据的ID和分表映射记录下来

分库分表产生的问题

  1. 查询数据难度增加,需要先查询映射表,再查询到数据
  2. 分布式ID. UUID
  3. 分布式事务, 之前的本地事务没法用了
    1. 在业务里避免分布式事务
    2. 用分布式框架
  4. 跨表关联/分页/排序: 在业务代码中分别查询,然后组装数据 Sharding-Proxy
  5. 业务服务器直连, 数据库(代理). 都是把逻辑抽象封装,业务无需关心细节

分库分表如何落地?

Mybatis

Mybatis我们项目用的是MybatisGenerator,尽量不写SQL,在Java代码中解决.
比如:用一张新表生成统计数据,定时执行.或者直接把数据存入Elasticsearch中,解决搜索和统计的问题.

一级缓存和二级缓存的区别

一级缓存范围在一个SqlSession内部,不是全局缓存
二级缓存是全局
如果二级缓存开启的话,执行顺序是先去二级缓存查询,二级缓存有,则直接返回.没有再去一级缓存查,有直接返回.没有的话,去数据库查.
多表查询的时候,多个sqlSession的时候,另外一个事务未提交的时候,二级缓存不生效.
一个事务已经提交的时候,二级缓存生效有50%的概率.
使用微服务框架的话,二级缓存和一级缓存的变量都可能不在同一台服务器上,容易出现脏读~幻读的现象.不建议使用.

GC

GCRoot

GCRoot指的是栈中,本地方法栈,方法区.
直接引用的对象,或者间接引用的对象.
没有被引用则会被删除

GC算法

  1. 标记-清理: 标记后直接清理
    缺: 内存碎片
  2. 标记-整理: 标记后删除,再前移动
    缺: 移动频繁,开销大
  3. 复制: 复制到另一块区域
    缺: 内存开销大

实际GC

分为新生区,和老年区
新生区Young: 分为Edan, Survivor0, Survivor1. 因为很多对象生命周期短,所以是8:1:1
老年区Old: Old. 大对象直接出生在Old区,比如大数组.

实际GC:
新创建的对象实例,在Edan.然后Young GC. 一次Edan+S1复制到S0.再一次GC由Edan+S0复制到S1.每一次复制,年龄+1.
到Age=6的时候,复制到Old.
Old满了,产生FullGC.用的是标记清理或标记整理的算法.

  • 如果Edan+S1复制到S0满了,直接进入Old区

Java的多态是什么?

编译类型是一个类,实际类型可能不同(可以是当前类,也可以子类),实际会调用实际类型的方法,造成方法行为的不同.

Java中变量,代码块,构造器之间的执行顺序是怎样的?

先父子静态变量;再父子静态代码块;先父普通成员变量+代码块+构造器;再子普通成员变量+代码块+构造器.

final关键字有什么作用?

  • 用于类时:类不能被继承,且所有方法也被final修饰

  • 用于变量:变量一旦初始化,不能被修改

抽象类有那些特点?

  3. 抽象类可以被继承,子类没实现,则继续作为抽象类

基础

1.Redis是什么?

又基于内存,又持久化的键值对数据库.

2.Redis过期key是怎么样清理的?

  1. 惰性清理: 查询时发现过期了,将key删除
  2. 定时清理: 周期性地清理所有DB,从DB的expires字典(key对象,value过期时间)随机取出20个key,删除掉过期的.
    如果超过25%的话,就对这个db清理.没有就下一个.
  3. 内存不够时清理:
    1. 不处理,写入时报错
    2. 从所有key中挑选,进行淘汰.
      1. 随机挑选进行淘汰
      2. 根据使用时间最远的淘汰
      3. 使用频率最低的进行淘汰
    3. 从设置了过期时间的key中,进行淘汰.

4.Redis的性能为什么这么高?

  1. 完全基于内存,没有磁盘IO,速度快
  2. 单线程,没有上下文的切换开销,不用考虑加锁释放锁,也没有死锁
  3. 数据结构专门设计的
  4. 监听多个IO流,空闲时把线程阻塞掉.有事件时,把阻塞唤醒,轮询发出事件的流,按顺序处理流.
    单个线程处理多个连接请求,减少网络IO.

8.如何解决Redis缓存穿透问题?

故意请求一些Redis不存在的key,请求打到数据库上,导致数据库压力过大.

  1. 对传参进行校验,避免一些无效redis请求
  2. 对于找不到的key,设置个短期过期时间
  3. 把所有可能存在的数据放入到大的map中,通过map进行过滤

9.如何解决Redis缓存击穿问题?

某个热点key过期,导致大量请求转向数据库,数据库压力过大

  1. 热点key设置永不过期
  2. 对于热点数据加锁(加在数据库上),每次只能取一次,每次等待100ms.

10.如何解决Redis缓存雪崩问题?

短期内大量key失效,所有请求转向数据库

  1. 给缓存失效时间加随机值,避免集体失效
  2. 搞两个缓存,缓存B比缓存A长一点,如果缓存A失效了,去缓存B找数据,然后用个异步线程来更新缓存A

11.如何解决缓存与数据库的数据一致性问题?

原因:

  1. 多个写请求的执行顺序不同导致脏数据.
  2. 先读了,再更新
  3. 数据库读写分离的,从库需要时间一定时间才能更新

错误解决方案:

  1. 先更新数据库,再更新缓存
  2. 先删除缓存,再更新数据
  3. 先更新数据库,再删除缓存

正确方案:

  1. 写请求之前要先获取分布式锁,才能更新,效率变低了
  2. 先更新数据库,异步删除缓存,删除失败后重试

持久化

1.Redis的持久化是怎么实现的?

将数据写入磁盘,有两种模式,AOF和RDB

  • AOF持久化

    把每次修改命令,添加到aofbuf缓存区的末尾,根据配置每秒写入AOF文件

  • RDB持久化

    RDB持久化:把这个时间点的所有键值对生成dump.rdb文件,替换旧的
    fork一个子进程,对键值对遍历,生成rdb文件

2.AOF和RDB的区别是什么?

AOF相当于操作日志,RDB相当于一次快照

3.怎么防止AOF文件越来越大?

周期性AOF重写命令,等于大到一定程度,就去掉一部分操作日志,重新定义初始值.
1+2+3+4+5+6->6+4+5+6

4.Redis持久化策略该如何选择?

RDB持久化时会阻塞Redis的主线程,有一定的IO压力
AOF持久化时aofbuf同步到磁盘有一定I/O压力,而且频率也高很多.

什么是关系型数据库?

建立在关系模型基础上的数据库,一对一,一对多,多对多的关系.

mysql事务

事务就是多个操作打包在一起,要么都执行,要么都不执行

ACID

  • A: 原子性: 事务是最小单位,要么全部完成,要么不起作用
  • C: 一致性: 执行事务前后,数据保持一致.(事务前后两位转账人总额不变,是目的)
  • I: 隔离性: 一个用户的事务不被其他事务所干扰
  • D: 持久性: 事务被提交后,对于数据的改变是持久的

索引类型

主键,唯一索引,普通索引,全文索引

索引优化有哪些建议

  1. 创建主键索引
  2. 创建唯一索引: 避免重复数据的插入,且提升查询效率
  3. 创建联合索引: 如果经常多个列一起查询的话. 注意把列的顺序,把更分散的列放在前面.
  4. 创建前缀索引: 对于文本类型的列,只对文本前面一部分进行索引
  5. 不要使用全文索引代替普通索引: 全文索引,不能用于排序和分组
  6. 避免使用过多的列创建索引: 索引列越多,索引的大小就越大,维护代价越高
  7. 避免含有Null值的列作为索引列: 含有NULL值的无法B-Tree索引

Mysql性能优化

  1. 慢SQL定位与分析
    • 使用Mysql慢查询日志
    • 使用Explain,分析下查询计划,索引的使用情况,是否有做全表扫描
  2. 索引~表结构和SQL优化
    • 索引优化:
      • 合理设计索引：在WHERE、JOIN、ORDER BY字段上创建索引
      • 复合索引顺序：如果经常多个列一起查询的话. 注意把列的顺序,把更分散的列放在前面.(最左前缀原则)
      • 避免过度索引：索引会降低写入性能
      • —定期维护索引：重建碎片化严重的索引
    • 表结构优化: 选择合适的字段类型~避免冗余字段~合适使用范式
    • SQL优化:
      • -----–—select
      • 避免使用SELECT *,只查询需要的字段
      • -----–—join
      • 减少join操作: 必要时冗余字段
      • 用连接代替子查询,减少查询次数
      • -----–—where
      • 查询条件不使用函数
      • 用union来代替or
      • 使用limit分页: 大数据量使用id > ? limit ? 代替 limit offset, size
  3. 架构优化
    • 读写分离: 读多写少时,可以做读写分离
    • 分库分表: 垂直分表,把大表分割成小表; 水平分表: 范围分表:按id范围,时间范围; 哈希分表
    • 数据冷热分离:
    • 缓存机制: 把热点数据缓存到内存中,redis

大数据量(超100万)的批量写会导致什么问题

  1. 阻塞其他操作: 占用CPU,内存,磁盘,系统响应慢
  2. 磁盘空间不足: 导致写操作失败
  3. 日志过大: 写操作时,mysql生成事务日志保证数据的一致性.会导致日志过大,写入速度变慢
  4. 死锁: 多个客户端批量写操作,操作数据范围重叠,可能会导致死锁
  5. 数据库性能下降:
  6. 主从延迟: 主从复制结构. 从服务器在处理写操作时遇到锁冲突,复制进程会被阻塞,导致主从延迟.

---------------–—措施

  1. 减少批量写的数据量, 分批进行写入
  2. 预留足够的磁盘空间和系统资源,确保能正常执行
  3. 优化表结构和索引, 提升写操作的性能和效率
  4. 使用事务进行批量写操作,确保数据一致性

Mysql修改大表的表结构会出现什么问题

  1. 长时间的阻塞: mysql会将表锁定,阻塞其他用户对表的读写操作,导致应用程序的延迟或响应时间变慢
  2. 需要大量的空间和时间: 修改大表结构,需要为该表创建新的临时表复制到临时表里.这个过程需要大量的空间和时间.
  3. 数据丢失: 发生错误,可能导致丢失或不一致,最好备份数据库
  4. 应用程序问题: 应用程序依赖表的结构,导致应用程序出bug

登录鉴权怎么实现?

  1. 用户提交用户名和密码
  2. 服务器验证并且生成JWT(JSON Web Token)
  3. 返回JWT给客户端
  4. 客户端存储JWT(放localStorage或Cookie里)
  5. 在Authorization中放JWT
  6. 服务器验证JWT签名和有效期

优点

  • 无状态,适合分布式系统
  • 存储在客户端上,减少服务端存储压力
  • 可以自定义数据

缺点

  • Token无法主动失效
  • Token过期需要自己去刷新

@Async: 实现异步方法调用

把任务分解成多个子任务,在后台或者其他线程去执行子任务,提高程序性能和响应能力

@Async简介

是Spring提供的注解,标记方法为异步执行的方法.告诉Spring调用这个方法时,放到线程池中异步执行,而不是等待方法完成.
---------------–—工作原理
调用被注解的方法时,Spring把方法的执行转移到线程池的一个线程中执行.
执行完成后,方法的返回值通过Future封装,获取返回结果.
---------------–—优势

  • 代码简化: 把异步执行的代码与业务逻辑分离
  • 网络请求时,快速响应,提高系统的并发处理能力
  • 耗时计算时,在后台执行计算过程,避免阻塞主线程,提高系统响应速度
  • 可以同时执行多个任务,节省处理时间

@Async源码解析

首先Spring要启用异步支持,Spring扫描容器的Bean,检查方法是否有@Async.
如果有@Async,把它封装成代理对象,实际是一个可执行的异步任务.
调用这个方法时,把这个异步任务放到线程池里去执行,返回Future对象.

异步方法与事务的关系

异步方法可以声明事务,但是事务只跟异步方法有关和主方法是无关.
需要非常注意逻辑的正确性,还有数据的一致性

Async异常处理

直接设置异步方法的返回参数为Future<String>,使用Future.get(),处理异常.

最佳实践和注意事项

  • 尽量简单,不涉及事务
  • 执行时间不要过长,导致线程资源占用太多
  • 注意和其他业务的一致,保证执行顺序和结果正确
